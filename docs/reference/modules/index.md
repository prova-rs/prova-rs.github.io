# Modules

Prova ships a set of first-party capability modules — the tools that bring a system into existence and poke it. They are **injected as globals** into every test file: no `require()` is needed, `shell`, `http`, `docker`, and friends are simply there.

```lua
local r = shell.run("cargo build --release", { check = true })
local res = http.get("http://127.0.0.1:8080/health")
```

Two kinds are built in, and the split is deliberate. **Primitives and substrate**: `shell`, `fs`, `net`, `docker` — the foundation everything composes on. **Network-drive clients**: `http`, `grpc`, `graphql` — how you *drive the app under test* — plus `yaml` (a parse utility) and `sqlite` (the one embedded, no-Docker database). Each heavyweight module is behind a **default-on Cargo feature** (`http`, `sqlite`, `docker`, `grpc`, `graphql`, `yaml` — feature name = namespace), so custom builds can opt out of a dependency tree; a standard prova binary has all of them. The `archetect` module is wired in by the prova host from the `prova-archetect` crate.

:::info Service clients and containers live in plugins
**Resource clients are not built in.** Databases, caches, brokers, object stores — every *containerized* service dependency — are **external plugins** (`prova-rs/prova-<name>`): pure-Lua modules declared under `[plugins]` in `prova.toml` and attached with `require()`. See the [official plugin catalog](../../plugins/official-plugins.md) for Postgres, MySQL, Redis, Kafka, Pulsar, RabbitMQ, and S3/MinIO, and [Plugins](../../plugins/index.md) for how the model works. This keeps the binary lean and privileges no technology.
:::

## The grammar

Every service namespace — built-in or plugin — has two doors. **`X.client(...)` attaches** to something already running. **`X.container(ctx, opts?)` provisions** an ephemeral instance via Docker and returns the **standard resource shape**:

| Member | Description |
|---|---|
| `client` | An open, already-managed client (present when the resource has a client) |
| `url` | The connection string that reaches the instance — the thing you inject into the app under test's environment |
| `container` | The underlying [container handle](docker.md#container) |
| `host` | The primary endpoint's host (`"127.0.0.1"`) — so env wiring is `DB_HOST = res.host` |
| `port` | The primary endpoint's mapped host port — `DB_PORT = res.port` |

Some resources carry extras (the s3 plugin's resource adds credentials). Learn the shape once and every resource — first-party or third-party — reads the same, because they are all generated by the same [`prova.containerized`](../../plugins/authoring-plugins.md#provacontainerized) scaffolding.

The network-facing clients are **plaintext-only in v1** — no TLS, no SASL, no token auth. They are aimed at localhost services and ephemeral CI containers, which is exactly what black-box acceptance tests stand up.

| Module | Purpose | Needs | Reference |
|---|---|---|---|
| `fs` | Read, write, glob, and inspect files; temp dirs | — | [fs](fs.md) |
| `shell` | Run commands (`shell.run`) and manage long-running processes (`shell.spawn`) | — | [shell & net](shell.md) |
| `net` | `net.free_port()` — an OS-assigned free TCP port | — | [shell & net](shell.md#net) |
| `http` | HTTP verbs, reusable REST clients, readiness polling | — | [http](http.md) |
| `grpc` | Native dynamic gRPC client via server reflection — no `.proto` files | Reflection-enabled server | [grpc](grpc.md) |
| `graphql` | GraphQL queries/mutations over HTTP POST | — | [graphql](graphql.md) |
| `docker` | Ephemeral containers (testcontainers-style) with readiness gates, exec, `container:run` | Docker daemon | [docker](docker.md) |
| `sqlite` | Embedded SQL — the one database that needs no container | — | [sqlite](sqlite.md) |
| `yaml` | Parse YAML text (including multi-document streams) to Lua values | — | [yaml](yaml.md) |
| `archetect` | Render and verify [Archetect](https://archetect.github.io) archetypes in-process | — | [archetect](archetect.md) |

Alongside the module globals, the `prova` global carries **`prova.parse.*`** (`lines`/`rows`/`table`/`json` — the CLI-output parsing toolkit) and **`prova.containerized`** (the scaffolding containerized resources are authored through); both are documented under [Authoring Plugins](../../plugins/authoring-plugins.md#the-docker-exec-toolkit).

Tests that need the Docker daemon — including any test that calls a plugin's `X.container` — should declare `requires = { "docker" }` so they **skip gracefully** (never fail) on machines where it is absent — see [Dependencies & Scheduling](../../writing-tests/dependencies-and-scheduling.md).

Module functions that perform I/O (`shell.run`, everything in `http`, `docker.run`, all the `client` and `container` functions) are **async under the hood** and must be called inside a fixture factory or test body, where prova's runtime is driving the coroutine. The `prova` and `ctx`/`t` surfaces they compose with are documented under the [Lua API](../lua-api/index.md).
